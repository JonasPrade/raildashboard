## Roadmap-Steuerung

Die Roadmap wird über ein **GitHub-Projects-Board** gesteuert — Regeln für die
Zusammenarbeit (Status-Wechsel, Issue-Format, User-Backlog, Release-Gate) stehen
in [`docs/github-projects.md`](docs/github-projects.md). `docs/roadmap.md` ist
nur noch Lese-Übersicht; lose Ideen kommen in `docs/user-backlog.md`.

## Alembic / Database Migrations

Use the Makefile commands for all Alembic interactions — never call `alembic` directly.

| Task | Command |
|---|---|
| Apply all pending migrations | `make migrate` |
| Create a new autogenerated revision | `make migrate-create MSG="describe the change"` |
| Run migrations inside Docker container | `make docker-migrate` |

## Deployment / Release (tag-basierte CI/CD)

Produktion läuft seit **v0.0.5** über eine tag-getriebene Pipeline
(`.github/workflows/deploy.yml`): **GitHub Actions baut** die Images und pusht sie nach
**GHCR**, der **Server zieht sie nur** (`ghcr.io/jonasprade/raildashboard-<svc>:${IMAGE_TAG}`)
— der Server baut nie selbst. Ein Release = ein Tag:

```bash
make release-check MILESTONE=vX.Y.Z          # Release-Gate (siehe AGENT.md)
# CHANGELOG.md: [Unreleased] → ## [vX.Y.Z] - YYYY-MM-DD verschieben, committen
git tag -a vX.Y.Z -m "…" && git push origin vX.Y.Z
```

Der Rest (Quality-Gates → Build → GHCR-Push → SSH-Deploy mit DB-Backup vor der Migration,
Health-Wait, Auto-Rollback) läuft automatisch. Regeln stehen in **AGENT.md** →
*Release & Deploy*, der vollständige Deploy-Vertrag in
[`docs/production_setup.md`](docs/production_setup.md) → *Deploy-Vertrag (tag-basierte CI/CD)*.

Konsequenzen für Änderungen:
- `docker-compose.yml` referenziert nur GHCR-Images (`image:`), **nie** `build:` — lokale
  Builds gehören in `docker-compose.override.yml`.
- `apps/backend/requirements.txt` bleibt exakt (`==`) gepinnt.
- Neue **Pflicht-Felder in `Settings`** (`core/config.py`) brauchen einen Default in
  `apps/backend/tests/conftest.py` (`os.environ.setdefault(...)`), sonst bricht der CI-Test-Gate
  auf einem sauberen Checkout ohne `.env`.
- Bei Deploy-relevanten Änderungen `CHANGELOG.md` **und** `docs/production_setup.md` nachziehen.
